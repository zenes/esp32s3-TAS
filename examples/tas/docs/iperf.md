# iPerf 사용 가이드 (tas 예제)

본 문서는 `tas` 예제에 추가된 iPerf 쉘 명령어의 사용법과 측정 방법을 한국어로 안내합니다. 본 iPerf 기능은 **LwIP 내장 `lwiperf` 기능(iPerf 2.x 호환)**을 기반으로 동작합니다.

## 1. 쉘에서 iPerf 서버 실행하기

ESP32 보드의 시리얼 터미널 창(Serial Monitor)에서 쉘을 통해 TCP 또는 UDP 서버를 실행하거나 중지할 수 있습니다.

### TCP 측정 명령어 (기본 대역폭 테스트)
- `iperf start`: iPerf TCP 서버를 수신 포트(5001)에서 백그라운드로 실행합니다.
- `iperf stop`: 실행 중인 iPerf TCP 서버를 강제로 종료합니다.

### UDP 측정 명령어 (커스텀 고속 수신 테스트)
- `udp_iperf start`: **초경량 UDP 수신 서버**를 수신 포트(5002)에서 실행합니다. 터미널에 1초마다 실시간 수신 속도가 계산되어 표시됩니다.
- `udp_iperf stop`: 실행 중인 UDP 수신 서버를 종료합니다.

**실행 예시:**
```text
> iperf start
iPerf TCP server started (Port 5001). Connect with: iperf -c <IP> -i 1 -t 10
> 
```

## 2. PC 또는 스마트폰에서 속도 측정하기

ESP32가 iPerf 서버로 동작하므로, PC 나 스마트폰에서 **iPerf 2 (또는 2.x) 클라이언트**를 이용하여 속도를 측정해야 합니다.
*(주의: iPerf3와는 호환되지 않으므로 반드시 iperf2를 사용하세요.)*

### 2.1. 측정용 IP 주소 확인하기
쉘에서 `ifconfig` 명령어를 입력하여 ESP32의 접속 IP를 확인합니다.
측정 환경에 따라 IP가 다를 수 있습니다:
- **이더넷 망(유선)에서 측정 시:** `[ETH]` 항목의 IP 주소 사용 (예: 192.168.0.50)
- **WiFi AP 망(무선)에서 측정 시:** `[AP]` 항목의 IP 주소 사용 (예: 192.168.4.1)

### 2.2. PC에서 클라이언트 실행 (명령프롬프트 / 터미널)
iPerf2를 다운로드 받은 폴더로 이동하여 아래 명령어를 실행합니다.

**[TCP 측정 시 (iperf start)]**
```bash
# 기본 측정 (10초간 측정)
$ iperf -c [ESP32의 IP 주소] -i 1 -t 10

# 측정 예시 (ESP32 IP가 192.168.0.50 인 경우)
$ iperf -c 192.168.0.50 -i 1 -t 10
```
- `-c`: 클라이언트 모드로 동작 (TCP)
- `-i 1`: 1초마다 중간 상태 출력
- `-t 10`: 총 10초 동안 대역폭 측정

**[UDP 측정 시 (udp_iperf start)]**
```bash
# UDP 모드로 20Mbps 속도로 강제 송신 (10초간 측정)
$ iperf -c [ESP32의 IP 주소] -u -p 5002 -b 20M -i 1 -t 10

# 측정 예시
$ iperf -c 192.168.0.50 -u -p 5002 -b 20M -i 1 -t 10
```
- `-u`: UDP 모드로 동작
- `-p 5002`: UDP 수신 포트를 5002번으로 지정
- `-b 20M`: PC에서 발생시킬 목표 대역폭 (20Mbps)을 설정 (UDP는 이 값을 주지 않으면 기본 1Mbps로만 아주 느리게 테스트하므로 필수입력)

## 4. 예상 측정 속도 (Throughput) 분석 및 기술적 한계
위의 테스트 결과를 종합하면 다음과 같습니다.

| 프로토콜 | 실측 속도 (Mbps) | 실측 속도 (MB/s) | 비고 |
| :--- | :--- | :--- | :--- |
| **TCP** | 약 14 ~ 15.5 Mbps | 약 1.7 ~ 1.9 MB/s | LwIP 스택 부하 포함 |
| **UDP** | **약 21.0 Mbps** | **약 2.5 MB/s** | 커스텀 경량 수신기 기준 |

UDP 속도가 TCP보다 더 높게 나오는 이유는, TCP는 패킷을 받을 때마다 확인 응답(ACK)을 보내고 순서를 맞추는 무거운 연산을 CPU가 실시간으로 수행해야 하지만, 저희가 구현한 UDP 커스텀 수신기는 단순히 들어오는 데이터를 카운트만 하기 때문에 Overhead가 훨씬 적기 때문입니다.

### 정상 속도 구간 안내
* **TCP:** 약 10 Mbps ~ 15 Mbps (대략 1.2 MB/s ~ 1.8 MB/s)
* **UDP:** 약 15 Mbps ~ 25 Mbps (대략 1.8 MB/s ~ 3.0 MB/s)
* 터미널 리포트 결과가 위 수치 범위 내에 있다면 하드웨어와 소프트웨어가 가장 이상적으로 최적화되어 동작하고 있는 상태입니다.
* 만약 5 Mbps 이하로 저하될 경우 SPI 클럭 설정, WiFi 간섭, 혹은 CPU 과부하 문제 등을 점검해 볼 필요가 있습니다.

### 속도 제한의 기술적 원인
현재 구조(W5500 + LwIP + NAPT)에서 15 Mbps 이상으로 속도를 올리기 어려운 주된 이유는 다음과 같습니다:

1. **내장 하드웨어 TCP/IP 엔진(TOE) 미사용 및 소프트웨어(LwIP) 오버헤드:**
   현재 코드는 브릿지 환경과 NAPT 기능을 구현하기 위해 LwIP 소프트웨어 스택과 ESP32 내장 Netif 구조를 의존하고 있습니다. 이로 인해 W5500의 속도에 강점을 갖는 '하드웨어 TOE(TCP Offload Engine)'을 직접 사용하지 못하고, 원시 패킷(MACRAW) 단위로 데이터를 올려 ESP32 CPU에서 소프트웨어(LwIP)적으로 패킷을 처리해야 하므로 변환 오버헤드가 크게 발생합니다.

2. **SPI 통신의 대역폭 병목:**
   W5500은 ESP32와 SPI(직렬) 통신으로 데이터를 교환합니다. MII/RMII(병렬) 인터페이스를 사용하는 다른 최상위 모델 이더넷 PHY(예: LAN8720)와 비교할 때 태생적으로 대역폭의 한계가 명확하게 존재합니다. (보통 SPI 클럭을 30~40MHz 이상으로 올려도 실효 전송률 한계에 부딪힘)

3. **중간 NAT(브릿지 연산) 및 이중 무선 송수신 로드:**
   순수하게 이더넷 케이블 하나만 연결된 환경이 아닙니다. `tas`는 내부적으로 **(수신) 이더넷 패킷 입력 -> (연산) ESP32 CPU (NAPT 처리) -> (송신) WiFi AP 출력** 이라는 3단계를 실시간으로 거쳐야 합니다. 이 과정에서 발생하는 CPU 연산 지연 및 무선(WiFi) 간섭 환경의 영향을 복합적으로 받기 때문에 속도는 더욱 반감됩니다.

## 5. 구간별 iPerf 종합 측정 결과 (End-to-End 테스트)

해당 테스트는 라즈베리파이(이더넷 접속), ESP32(브릿지 서버), PC(WiFi 접속) 간의 구간별 성능 및 병목점을 파악하기 위해 진행되었습니다.
* **테스트 시스템 구성 경로:** `Raspberry Pi (Client) -(eth)-> ESP32 (iPerf Server / Bridge) -(wlan)-> PC (Client)`

### 5.1. TCP 대역폭 측정 결과

| 테스트 구간 (Path) | 평균 대역폭 | 결과 분석 및 비고 |
| :--- | :--- | :--- |
| **순수 이더넷 구간**<br>`RSPI -(eth)- ESP32` | **16.9 Mbps** | 속도 저하(Drop) 하나도 없이 16.8Mbps로 극도로 안정적인 전송 유지.<br>3개 측정 구간 중 최고 속도 및 시스템 내 최고 안정성을 보임. |
| **순수 무선(WiFi) 구간**<br>`ESP32 -(wlan)- PC` | **12.8 Mbps**<br>*(안정 구간 13~15 Mbps)* | 간헐적인 속도 저하(출렁임)는 발생하나 연결 끊김이나 심각한 지연은 없음.<br>**해당 시스템 구성의 실질적인 메인 네트워크 병목 구간으로 확인됨.** |
| **단말 간 통합 브릿지 구간**<br>`RSPI -(eth)- ESP32 -(wlan)- PC` | **6.86 Mbps**<br>*(상위 15초 유지: 약 10 Mbps)* | 테스트 초기 15초 동안 9.5~10.5Mbps의 대역폭을 안정적으로 유지함.<br>단, 장시간 지속 전송 시 버퍼 및 NAPT 처리 부하 누적으로 전체 평균 대역폭은 하락. |

### 5.2. UDP 스트리밍 측정 결과

| 테스트 구간 (Path) | 평가 설정 (부하량) | 평균 대역폭 | 결과 분석 및 비고 |
| :--- | :--- | :--- | :--- |
| **순수 이더넷 구간**<br>`RSPI -(eth)- ESP32` | 10M, 20M, 30M 부하 | **30+ Mbps** | 30Mbps의 거센 부하를 주어도 전송 지연이나 버퍼 포화 현상이 없이 매우 안정적으로 전송됨.<br>이더넷 링크 자체의 여유 능력이 무선 구간 및 브릿지 라우팅 한계치보다 훨씬 넓음을 확인. |
| **순수 무선(WiFi) 구간**<br>`ESP32 -(wlan)- PC` | 10M, 20M, 30M 부하 | **약 15~20 Mbps** | 10Mbps는 완벽하게 소화하나, 20Mbps 이상 전송 시 PC 측 송신 버퍼 포화(`No buffer space available`) 및 전송 속도 급감 발생.<br>**WiFi 무선 링크의 물리적 한계 및 임계점이 15~20Mbps 사이임을 증명함.** |
| **단말 간 통합 브릿지 구간**<br>`RSPI -(eth)- ESP32 -(wlan)- PC` | 10M 부하 | **약 10 Mbps**<br>*(서버 수신 8.52 Mbps)* | 10Mbps 부하 테스트에서 패킷 지연(Jitter) 매우 안정적, **Data Loss 0%** 기록.<br>전체 End-to-End 라우팅 과정을 거치면서도 성능 누수 없이 안정적인 통신을 보장하는 최적의 제한폭임. |

## 6. FAQ: iPerf 속도와 실제 파일 복사(SCP, FTP) 속도의 차이

iPerf에서 측정한 속도(약 1.8~1.9 MB/s)보다 SSH를 통한 SCP나 SFTP 파일 복사 속도(약 1.0 MB/s 내외)가 현저히 낮게 나오는 것은 정상적인 현상입니다.

1. **iPerf의 특성 (가장 가벼운 더미 데이터 전송):**
   * iPerf는 패킷 내부에 의미 없는 쓰레기(Dummy) 데이터를 꽉 채워서 연속적으로 밀어 넣고, 도착 여부만 순수하게 체크합니다. 순수한 원시 "네트워크 파이프 굵기" 자체를 잴 뿐입니다.
   * 저장 장치(SD카드/Flash) 접근이나 암호화 연산이 전혀 없습니다.

2. **SCP (Secure Copy)의 특성 (매우 무거운 환경):**
   * **막대한 암호화 오버헤드:** SCP/SSH는 오가는 **모든 패킷 데이터를 실시간으로 고도의 암호화(Encryption)** 및 복호화(Decryption) 처리를 합니다. ESP32와 같은 임베디드 소형 CPU에게 이 암호화 연산은 네트워크 전송보다 훨씬 버거운 작업입니다. 결국 네트워크 파이프는 널널해도 CPU가 암호화를 계산하느라 속도를 못 따라갑니다. (이것이 가장 큰 원인입니다.)
   * **파일 시스템 접근 지연 (Flash/SD카드):** 네트워크에서 받은 데이터를 실제 메모리(또는 SD/Flash)에 기록해야 하므로 디스크 I/O 속도의 제한도 함께 받습니다.
   * **TCP 윈도잉/ACK 검증 지연:** 데이터를 안전하게 파일로 조립하기 위해 패킷 조각을 확인하고 승인(ACK)하는 복잡한 과정이 추가됩니다.

**결론:** iPerf는 **"고속도로(네트워크)의 최대 제한 속도(1.9 MB/s)"**를 측정한 것이고, SCP 파일 전송은 **"검문소(암호화 처리 CPU)를 거쳐서 화물을 내리는 트럭의 실제 체감 속도(1.0 MB/s)"**가 되는 구조적 차이입니다. 암호화가 없는 일반 FTP(또는 HTTP)를 사용하시면 iPerf에 조금 더 근접한 속도를 얻으실 수 있습니다.
