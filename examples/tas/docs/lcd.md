# ESP32-S3 및 LilyGO T-ETH-Lite 보드에서의 LCD 구동 방식 분석

본 문서는 ESP32-S3 기반 보드(특히 LilyGO T-ETH-Lite-ESP32S3) 환경에서 디스플레이(LCD)를 구동할 때 발생할 수 있는 인터페이스별 성능 차이(SPI, RGB 병렬, 8-bit 병렬)와 하드웨어 핀 제약 조건을 정리한 것입니다.

## 1. 디스플레이 인터페이스 성능 비교: SPI vs RGB (16-bit 병렬)

화면을 그릴 때 **RGB 인터페이스(병렬)가 SPI(직렬) 방식보다 압도적으로 빠릅니다.**

*   **SPI (Serial Peripheral Interface)**: 직렬 통신. 최대 약 80MHz까지 설정 가능. 체감 대역폭은 보통 30~40Mbps 수준.
*   **RGB (Parallel)**: ESP32-S3의 전용 LCD/CAM 하드웨어와 DMA를 사용해 CPU 개입 없이 비디오 메모리를 화면에 병렬(보통 16가닥)로 전송. 16배의 대역폭 확보. 최고 640Mbps (40MHz 클럭 기준).

### 1.1. 해상도별 이론 및 체감 프레임 레이트 (FPS)

*(조건: 1픽셀 = 16-bit RGB565 컬러)*

| 해상도 | SPI (40MHz 안정성 기준) | SPI (80MHz 한계치) | RGB (40MHz 기준) | 실 구동 체감 비교 |
| :--- | :--- | :--- | :--- | :--- |
| **320x240 (QVGA)** | 이론 32.5 FPS / 실제 20~25 FPS | 이론 65.1 FPS / 실제 45~50 FPS | 520+ FPS | **[SPI 권장]** TFT_eSPI의 Sprite+DMA 활용 시 30 FPS 방어 가능 |
| **480x320 (HVGA)** | 이론 16.2 FPS / 실제 10 FPS 내외 | 이론 32.5 FPS / 실제 20~25 FPS | 260+ FPS | **[RGB 권장]** SPI로는 LVGL 같은 화려한 UI 구동 시 끊김 심함 |
| **800x480 (WVGA)** | 6.5 FPS (사용 불가 수준) | 13.0 FPS (사용 불가 수준) | 104+ FPS | **[무조건 RGB 사용]** 부드러운 60 FPS 스크롤 및 애니메이션 가능 |

*   **TFT_eSPI의 30FPS 방어**: 320x240 해상도에서 TFT_eSPI 라이브러리로 30fps를 내려면 화면을 메모리에 미리 그려놓고(Sprite Buffer) 한 번에 전송(DMA)하는 기술의 사용이 필수적입니다. 화면 크기가 480x320 이상으로 커지면 SPI 방식으로는 물리적으로 30fps 구동이 불가능합니다.

## 2. API 상의 Clock 속도 설정 방식 차이

*   **SPI 인터페이스**: 아무 주파수나 입력해도 내부적으로는 APB 클럭(80MHz)을 정수로 나눈 값(Divisor)으로 자동 보정됩니다. 자주 쓰이는 권장 픽셀 주파수는 **40MHz, 26.6MHz, 20MHz** 등입니다. (80MHz는 노이즈에 극도로 취약함).
*   **RGB 인터페이스**: 패널 사양서(Datasheet)에 명시된 "Typical Pixel Clock (PCLK)"에 맞춰 10MHz~40MHz 사이의 임의의 Hz 단위 숫자를 자유롭게 입력합니다. 시스템이 PLL을 분주하여 가장 가까운 세밀한 클럭 전송(예: 480x272는 보통 9MHz, 800x480은 33MHz 등).

## 3. LilyGO T-ETH-Lite-ESP32S3 하드웨어 핀(Pin) 제약 분석

이더넷과 SD 카드를 동시에 지원하는 본 프로젝트의 보드 특성상, 디스플레이용으로 외부로 뺄 수 있는 여유 GPIO 핀의 개수에 극심한 제약이 따릅니다.

### 3.1. 16-bit RGB 디스플레이 연결 불가 사유
*   **요구 핀 수**: 최소 **20개** (데이터 16핀 + 제어 4핀).
*   **잔여 핀 수**: 이더넷 전용 W5500 SPI 통신핀, 내부 USB 핀, SD 카드 공유 핀, 전원 핀을 제외하고 확장 헤더(P2, P3, P4)로 꺼내어 온전히 쓸 수 있는 여유 GPIO는 최대 **18~19개**뿐입니다.
*   **결론**: 물리적인 데이터 핀 개수 부족으로 연결 자체가 **불가능**합니다.

### 3.2. 8-bit 병렬 (Parallel) 디스플레이 연결 (SD카드 미사용 + 스트래핑 핀 제외 조건)
만약 **"SD 카드를 포기하고"**, **"부팅(Strapping) 핀을 완전히 배제"**한다는 엄격한 전제 조건을 둔다면 **8비트 병렬 구동은 완벽하게 실현 가능**합니다.

*   **요구 핀 수**: 최소 **12~13개** (데이터 8핀 + 제어 4~5핀).
*   **잔여 안전 핀(Safe Pins)**: 확장 헤더(P2, P3, P4) 및 SD 슬롯 위치 핀 중에서 W5500 전용 핀과 부팅 핀(`IO0`, `IO45`, `IO46`, `IO3`)을 제외하면 아래와 같이 **총 18개의 완전히 안전한 General Purpose GPIO**를 확보할 수 있습니다.
    *   **P3 헤더**: `IO1`, `IO2`, `IO21`, `IO38`, `IO39`, `IO40`, `IO41`
    *   **P2 헤더**: `IO4`, `IO8`, `IO15`, `IO16`, `IO17`, `IO18`
    *   **P4 헤더**: `IO47`, `IO48`
    *   **SD 카드 슬롯 위치**: `IO5`, `IO6`, `IO7`
*   **실현 가능성 및 단점**:
    위 18개의 핀 중 12~13개를 지정(User_Setup.h 등)하여 디스플레이를 구동하면 스트래핑 핀 충돌(부팅 불가, 먹통 현상) 없이 **화면을 켤 수 있으며**, SPI에 비해 8배의 대역폭으로 한층 부드러운 60FPS(HVGA 해상도 기준) 방어가 가능해집니다.
    하지만 18개의 핀 중 13개를 디스플레이 하나에 소진하므로, 터치패널(I2C 등)이나 다른 센서를 연결하기 위한 **예비 핀이 약 5개밖에 남지 않아 시스템 확장성이 극도로 제한**된다는 점을 설계 시 반드시 감안해야 합니다.

## 4. 디스플레이 제어 핀(Control Pins) 및 메모리 구조의 이해

8비트 병렬 및 전용 디스플레이 모듈을 사용할 때 마주치는 주요 통신 핀들의 역할과 구동 원리를 정리합니다.

### 4.1. 핵심 제어 신호들 (8080 MCU 인터페이스 기준)
*   **TFT_CS (Chip Select)**
    *   **역할**: "이 칩 활성화 스위치". 이 핀이 `LOW`로 떨어졌을 때만 디스플레이 컨트롤러가 들어오는 데이터를 수신합니다.
    *   **특징**: 여러 칩이 데이터 선을 공유할 때 타겟을 식별하는 용도이며, 1개의 디스플레이만 전담으로 사용할 경우 보드의 `GND`에 영구 직결(Hardwired)하여 ESP32에서 GPIO 1개를 절약할 수도 있습니다.
*   **TFT_RS / TFT_DC (Register Select / Data or Command)**
    *   **역할**: 데이터 버스(D0~D7)에 실린 8비트가 **"명령어(Command)"**인지, 모니터에 뿌릴 **"색상 데이터(Data)"**인지 구분해 주는 신호등입니다.
    *   **주의**: 이름이 비슷해서 흔히 오해하지만 **진짜 장치 초기화용 리셋 핀(RST 또는 RESET)**과는 기능이 완전히 다릅니다.
*   **TFT_WR (Write)**
    *   **역할**: 데이터를 디스플레이 쪽으로 쑤셔 넣는 **"셔터 스위치(클럭)"** 역할.
    *   **동작 원리**: ESP32가 8가닥 선에 신호를 세팅한 뒤 `TFT_WR` 핀을 `LOW`에서 `HIGH`로 탁 튕겨주는 순간, 디스플레이가 그 값을 자신의 내장 메모리(GRAM)로 '찰칵'하고 빨아들입니다.

### 4.2. 내장 메모리(GRAM)가 있는 똑똑한 모듈 (SPI, 8-bit 병렬)
우리가 사용하는 **ILI9341, ST7789, ILI9488** 등의 모듈은 칩 내부에 화면 해상도 픽셀 크기만큼의 **그래픽 전용 RAM(GRAM)**을 품고 있습니다.
*   **장점 (ESP32의 휴식)**: ESP32가 화면이 바뀔 때만 `TFT_WR` 박자에 맞춰 데이터를 전송해주면 끝납니다. 평소에는 통신을 아예 끊어도 디스플레이가 혼자서 메모리에 담긴 화면을 유지하므로 ESP32의 CPU 리소스가 아주 여유로워집니다.
*   **단점 및 해상도 한계**: 칩 내부에 비싼 메모리를 구겨 넣어야 해서 보통 **480x320(HVGA)** 해상도까지만 제조됩니다.

### 4.3. 내장 메모리가 없는 깡통 디스플레이 (RGB 인터페이스)
화면 크기가 커지면(예: 800x480 이상) 내부에 1MB가 넘어가는 RAM을 넣을 수 없어 디스플레이 자체가 '메모리 없는 깡통'으로 만들어집니다.
*   **DE (Data Enable) 및 PCLK의 등장**: 내장 메모리가 없기 때문에 ESP32가 화면 데이터를 초당 60프레임씩 쉬지 않고 **비디오 스트리밍**하듯 부어주어야 압니다. 이때 *"지금부터 모니터 유효 영역에 찍히는 진짜 데이터다!"*라고 실시간으로 알려주는 동기화 신호가 바로 **DE(Data Enable)**와 **PCLK(픽셀 클럭)**입니다. (8080 병렬의 `TFT_WR` 셔터 방식과는 완전히 다름).
*   **단점**: ESP32가 화면 갱신을 위해 쉴 새 없이 CPU나 전용 DMA 하드웨어를 가동해야 하고 핀 전송량(20핀 이상)이 극심합니다.

## 5. 최종 권장 사항

이더넷 모듈(W5500) 및 SD 카드를 운용하는 특수 목적의 환경(LilyGO T-ETH-Lite-ESP32S3)에서는, 해상도를 320x240 내외로 유지하고 **핀을 적게 차지하는 SPI 인터페이스(TFT_eSPI 이용, 핀 4~6가닥 사용) 방식의 디스플레이를 활용하는 것이 시스템 안정성과 확장성 측면에서 제일 권장되는 방법**입니다.

## 6. T-ETH-Lite-S3 SPI LCD (ST7789) 실전 구동 가이드

프로젝트 진행 과정에서 확인된 최적의 설정값입니다.

### 6.1. 하드웨어 연결 (SPI2 / FSPI 사용)
이더넷(W5500)과의 SPI 버스 충돌을 방지하기 위해 반드시 **SPI2_HOST (FSPI)**를 사용해야 합니다.
- **SCK**: IO2, **MOSI**: IO3, **MISO**: IO1, **CS**: IO4
- **DC**: IO8, **RST**: IO15, **BL**: IO16

### 6.2. 소프트웨어 설정 (PlatformIO / LVGL 8.3)
색상 뒤바뀜(Color Swap) 및 글자 깨짐 현상을 해결하는 핵심 플래그입니다.

#### `platformio.ini` 설정
```ini
build_flags = 
    -DST7789_DRIVER
    -DTFT_RGB_ORDER=TFT_BGR  ; Red-Blue 스왑 및 색깔 밀림 해결
    -DLV_COLOR_16_SWAP=1     ; LVGL 버퍼와 LCD 사이의 엔디안(Endian) 정합성 해결 (글자 깨짐 방지)
    -DLV_TICK_CUSTOM=1       ; 전용 타이머 대신 millis() 자동 사용 (애니메이션 정상 작동)
    -DLV_USE_PERF_MONITOR=1  ; LVGL 내부 FPS/CPU 측정 기능 활성화
```

#### `tas.ino` 디스플레이 플러시 (Display Flush)
`LV_COLOR_16_SWAP=1`을 사용할 경우, `TFT_eSPI`의 `pushColors`에서 별도의 바이트 스왑을 수행하지 않아야 합니다.
```cpp
tft.pushColors((uint16_t *)&color_p->full, w * h, false); // swap 파라미터를 false로 설정
```

### 6.3. 성능 및 검증 (최상위 최적화 결과)
- **CPU 클럭**: **240MHz** (S3 최대 성능 모드).
- **SPI 클럭**: **80MHz** (하드웨어 한계치).
- **버퍼 구성**: **Double Buffering (160 Lines x 2)** - 내부 SRAM 활용 (75KB * 2).
- **기술 적용**: **SPI DMA (Non-blocking)** - 렌더링과 전송을 병렬로 처리.
- **실측 성능**: 
    - **부분 업데이트**: **80+ FPS** (매우 부드러움).
    - **전체 화면 업데이트**: **30 FPS 고정** (Flicker/Scanline 최소화).
- **색상 검증**: 하단 R, G, B 박스를 통해 정확한 색상 출력 확인 완료.

### 6.4. FPS 측정 메커니즘의 이해 (Partial vs Logical)

프로젝트 초기에는 모든 SPI 전송 횟수를 카운트했으나, 보다 정확한 성능 지표를 위해 LVGL 내부의 평균 FPS 계산 방식으로 전환했습니다.

| 구분 | Partial Flush (초기 방식) | Logical FPS (최종 방식) |
| :--- | :--- | :--- |
| **측정 대상** | `my_disp_flush` 호출 횟수 | LVGL 렌더링 주기 (완성된 프레임) |
| **수치 특징** | 80+ FPS (부분 갱신이 많을수록 높음) | 약 30 FPS (실제 시각적 프레임) |
| **정확도** | 전송 대역폭 확인용 | 실제 체감 성능 확인용 |
| **함수/플래그** | 수동 `frame_cnt++` | `lv_refr_get_fps_avg()` / `LV_USE_PERF_MONITOR` |
